## What is Curateur?

A personal dashboard delivered as a Telegram Mini App. It's the control panel for [OpenClaw](https://github.com/AiClaw/openclaw), a self-hosted AI agent framework. Single-user, not multi-tenant.

## Architecture

**Manifest-driven plugin system with three tiers:**

| Tier | Internal Type | Display Name (configurable) | Purpose |
|------|--------------|---------------------------|---------|
| View | `view` | Facets | Visual dashboard widgets |
| Action | `action` | Hooks | Commands that execute agent scripts |
| Connector | `connector` | Taps | MCP server config for OpenClaw |

Every plugin lives in `src/plugins/<name>/` with a `manifest.json` as the source of truth.

## Tech Stack

- **Frontend:** React 18, TypeScript, Vite 5, Lucide icons, vanilla CSS with Telegram theme variables
- **Backend:** Hono (Node.js), served via `@hono/node-server`, executed with `tsx`
- **Storage:** No database. Flat files on disk at `$WORKSPACE_DIR` (default `/root/.openclaw/workspace`)
- **Package manager:** pnpm

## Plugin Anatomy

### View Plugin (e.g., `src/plugins/exchange-rate/`)

```
manifest.json   â†’ { type: "view", widget: { component, defaultEnabled }, api?: { routes, prefix } }
widget.tsx      â†’ React component (default export, no props)
routes.ts       â†’ Hono router (default export), mounted at api.prefix
```

### Action Plugin (e.g., `src/plugins/new-session/`)

```
manifest.json   â†’ { type: "action", skill: { agent, command, inputs? } }
```

`skill.agent` references a shell script at `$WORKSPACE_DIR/agents/<name>.sh`. Actions are executed by sending the `skill.command` to Telegram Bot API.

### Connector Plugin (e.g., `src/plugins/mcp-filesystem/`)

```
manifest.json   â†’ { type: "connector", mcp: { serverName, configSchema?, defaultConfig? } }
```

Connectors read/write OpenClaw's MCP config at `~/.openclaw/openclaw.json` (JSON5). OpenClaw hot-reloads on file change.

## How to Add a Plugin

1. Create `src/plugins/<name>/manifest.json`
2. For views: add `widget.tsx` (React component)
3. For API-backed plugins: add `routes.ts` (Hono router)
4. Import the manifest in `src/plugins/registry.ts` and add to the `plugins` array

The registry drives everything â€” FacetSelector, HookRunner, TapManager, CommandPalette, and Settings all consume from it. No other registration needed.

## Backend

Hono server at `api/server.ts`:
- Plugin routes auto-mounted at manifest-declared prefixes
- Shared endpoints: `/api/config`, `/api/health`, `/api/message`, `/api/skill/:id/execute`, `/api/mcp`
- Shared libraries in `api/lib/`: `workspace.ts` (file I/O), `telegram.ts` (Bot API), `shell.ts` (exec)
- CORS middleware in `api/middleware/cors.ts`

## Frontend

- Shell components in `src/shell/` consume the plugin registry
- `FacetSelector` lazy-loads view widgets via `import()`
- `HookRunner` auto-generates forms from action manifest `inputs[]`
- `TapManager` reads/writes OpenClaw MCP config via `/api/mcp` endpoints
- `CommandPalette` dynamically builds search from all plugins
- Hooks: `usePlugin` (scoped fetch), `useSettings` (enabled plugins), `useNamingPack` (theme)
- Settings fetch from `/api/config` first, fall back to `localStorage`

## Config System

Two config files:
- **`.env`** â€” server secrets (tokens, paths, ports). Generated by `pnpm setup`
- **`curateur.config.json`** â€” app preferences (naming pack, enabled plugins). Editable via CLI setup or in-app Settings

## Conventions

- No external state management (no Redux/Zustand) â€” hooks + localStorage + server config
- Inline styles using Telegram CSS variables (`var(--c-text)`, `var(--c-bg)`, etc.)
- Plugin widgets are self-contained â€” no shared props, each fetches its own data
- Icons referenced by Lucide name strings in manifests, resolved via `LucideIcons[name]`
- All API routes return JSON
